Server / Redis Cache

What is Server-Side Caching?

Server-side caching stores computed results in a fast key-value store (like Redis or Memcached) that sits between your application and its data sources. When a request arrives, the server checks the cache first. If the data is there (a HIT), it skips the expensive computation or database query. If not (a MISS), it fetches the data, stores a copy in the cache for next time, and returns it.

Redis is the most popular server-side cache. It runs as a separate process, supports multiple data structures, and can be shared across multiple application instances — making it ideal for horizontally scaled deployments.

How It Works

The request flows through the server cache layer before reaching the origin data source:

{{ flow_diagram(["Client Request", "App Server", "Redis Cache", "Origin / DB"], "Redis Cache") }}
  1. Check cache: The app calls cache.get(key) — Redis looks up the key in memory.
  2. HIT path: If found and not expired, Redis returns the serialized value. The app deserializes and responds immediately.
  3. MISS path: If not found, the app fetches from the origin (database, API, computation), serializes the result, and calls cache.set(key, value, ttl).
  4. TTL expiry: After the TTL (time-to-live) elapses, Redis automatically deletes the key, ensuring stale data is eventually refreshed.
