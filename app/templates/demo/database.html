Database Cache

What is Database Caching?

Database caching stores the results of expensive database queries so they can be returned instantly on subsequent identical requests. Instead of executing a complex SQL query with joins, aggregations, or full-table scans every time, the result set is cached after the first execution.

This can happen at multiple levels: the database engine's internal query cache, an ORM-level cache, or an application-level cache layer that wraps database calls. The goal is always the same — avoid repeating work the database has already done.

How It Works

Database caching intercepts queries before they reach the database engine:

{{ flow_diagram(["Application", "Query Cache Layer", "Database Engine", "Disk Storage"], "Query Cache Layer") }}
  1. Query arrives: The application (or ORM) generates a database query.
  2. Cache lookup: A hash of the query (+ parameters) is used as the cache key. The cache layer checks for a stored result.
  3. HIT: Cached result is returned directly, bypassing the database entirely.
  4. MISS: Query executes against the database. The result is stored in cache with a TTL.
  5. Invalidation: When underlying data changes (INSERT, UPDATE, DELETE), related cache entries must be invalidated to prevent serving stale data.
