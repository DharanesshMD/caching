Browser Cache

What is Browser Caching?

Browser caching is the first layer of the caching stack. The browser stores copies of responses locally on the user's device, controlled entirely by HTTP headers sent by the server. When the browser has a cached copy, it can serve it instantly without making any network request at all.

This is the fastest possible cache layer because the data never leaves the user's machine — there's zero network latency. The trade-off is that the server doesn't control when the browser evicts cached data, and each user has their own independent cache.

How It Works

Browser caching is driven by HTTP response headers. The server tells the browser how long to keep a response and how to validate it:

{{ flow_diagram(["Browser checks local cache", "Cache fresh?", "Yes: serve from disk", "No: send request with ETag", "Server: 304 Not Modified?", "Serve cached / new response"], "Browser checks local cache") }}
  1. First request: Browser has no cache. Server responds with Cache-Control: max-age=3600 and ETag: "abc123".
  2. Subsequent request (within max-age): Browser serves directly from local cache — no network request at all.
  3. After max-age expires: Browser sends a conditional request with If-None-Match: "abc123".
  4. 304 Not Modified: If the resource hasn't changed, the server responds with just headers (no body), saving bandwidth.
  5. 200 with new body: If changed, the server sends the full new response with updated ETag.
