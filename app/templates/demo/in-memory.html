In-Memory Cache

What is In-Memory Caching?

In-memory caching stores data directly in the application process's memory (RAM). Unlike Redis or Memcached which run as separate services, an in-memory cache lives inside your application — typically as a Python dictionary with expiration logic.

This is the fastest server-side cache layer because there is zero network overhead. Data is accessed via direct memory lookups, measured in microseconds rather than milliseconds. The trade-off is that the cache is local to each process: it can't be shared across multiple application instances, and it's lost when the process restarts.

How It Works

In-memory cache is the closest layer to your application code:

{{ flow_diagram(["Application Code", "In-Memory Cache (dict)", "Redis / External Cache", "Database / Origin"], "In-Memory Cache (dict)") }}
  1. Lookup: The app checks an in-process dictionary for the key. This is a hash table lookup — O(1) average.
  2. HIT: Value is returned immediately from RAM. No serialization, no network.
  3. MISS: Falls through to the next cache layer (Redis) or origin (database).
  4. TTL & cleanup: Each entry has an expires_at timestamp. Expired entries are lazily cleaned up on access.
  5. Eviction: When memory is constrained, an eviction policy (LRU, LFU) removes the least useful entries.
